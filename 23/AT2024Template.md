# 1. Введение
Данный документ спецификации требований к программному обеспечению (SRS) описывает функциональные и нефункциональные требования к системе, обеспечивающей сервис доставки непродовольственных товаров. Система позволит пользователям выбирать товары из наличия, оформлять заказы, осуществлять оплату и заказывать доставку с учетом специфических требований.

## Цели
Цель данного документа заключается в:
- Предоставлении четкого и структурированного описания функциональных и нефункциональных требований к системе доставки.
- Упрощении процесса разработки и тестирования программного обеспечения.
- Создании основы для дальнейшего анализа и уточнения требований от конечных пользователей.

## Границы применения
Этот документ распространяется на программное обеспечение, которое включает в себя:
- Интерфейс для пользователей, позволяющий выбирать товары из склада.
- Модуль обработки заказов, включая оплату и выбор параметров доставки.
- Систему управления доставками для курьеров, включающую возможность указания этажа и квартиры для массивных товаров.

## Термины и аббревиатуры
- Система — программное обеспечение, обеспечивающее сервис доставки непродовольственных товаров.
- Пользователь — лицо, использующее систему для выбора и заказа товаров.
- Курьер — лицо, ответственное за доставку товаров пользователю.
- Заказ — запрос пользователя на доставку определенного количества товаров с указанными параметрами.
- Этаж — номер этажа, на который необходимо доставить товар. 
- Квартира — номер квартиры, в которую требуется доставить товар.

# 2. Общее описание

## Описание изделия
Система доставки непродовольственных товаров предназначена для упрощения процесса выбора и заказа товаров, а также организации доставки по указанным адресам. Она включает в себя следующие компоненты:
- Интерфейсы системы: 
  - Веб-интерфейс для пользователей.
  - Административный интерфейс для обработки заказов и управления товарными позициями.
  - Интерфейс для курьеров для получения информации о доставках.

- Интерфейсы пользователя: 
  - Веб-интерфейс с возможностью регистрации/авторизации, выбора товаров, оформления заказа и управления заказами.
  - Мобильное приложение (возможности на этапе разработки).

- Интерфейсы аппаратных средств ЭВМ: 
  - Сервер, на котором будет развернута база данных и веб-приложение.

- Интерфейсы программного обеспечения: 
  - База данных для хранения информации о товарах, пользователях и заказах.
  - API для интеграции с платежными системами.

- Интерфейсы коммуникаций:
  - Интернет-соединение для доступа к веб-приложению и API.

- Ограничения памяти:
  - Объем данных о пользователях, товарах и заказах должен нормально обрабатываться с учетом размеров базы данных.

- Действия:
  - Процесс выбора товаров, оформления заказа, оплаты и организации доставки.

## Функции изделия
- Основные функции:
  - Регистрация и авторизация пользователей.
  - Просмотр доступных товаров на складе.
  - Оформление заказа с указанием времени и адреса доставки.
  - Оплата заказа.
  - Информирование курьеров о деталях заказов.

## Характеристики пользователей
- Основные предусмотренные роли:
  - Пользователь: осуществляет покупки, управляет заказами.
  - Администратор: управляет товарными позициями и заказами.
  - Курьер: получает информацию о доставках.

## Ограничения
- Система должна обеспечивать безопасность данных пользователя.
- Должна быть обеспечена доступность сервиса 24/7.

## Предположения и зависимости
- Ожидается наличие стабильного интернет-соединения для работы системы.
- Зависимость от сторонних платежных систем для обработки платежей.

# 3. Детальные требования

## Функциональные требования
### 1. Заказ и управление им

1.1 Регистрация заказа:
- Платформа должна обеспечивать пользователям возможность оформлять заказы на доставку товаров через онлайн-интерфейс с доступом к актуальному ассортименту на складе.

1.2 Уникальный идентификатор заказа:
- Каждый новый заказ должен автоматически получать уникальный номер для удобства отслеживания.

1.3 Использование предустановленных шаблонов:
- Клиенты должны иметь возможность создавать заказы по заранее заданным шаблонам, которые включают такие данные, как адрес доставки и предпочтительное время.

1.4 Классификация и метки:
- Заказы должны поддерживать возможность классификации по категориям (например, по товарным группам) и добавления меток для более удобного поиска и фильтрации.

1.5 Валидация данных заказа:
- Система должна проверять введенные данные на корректность при создании заказа и подтверждать заполнение обязательных полей (например, адрес и время доставки).

1.6 Модификация заказа:
- Пользователь может изменять информацию о заказе до его подтверждения, включая коррекции адреса и времени.

1.7 Удаление незакрепленного заказа:
- Клиенты должны иметь возможность отменять заказы, которые еще не были подтверждены.

1.8 Прикрепление документов:
- В процессе оформления заказа пользователи могут добавлять файлы, такие как изображения или документы, для уточнения информации.

### 2. Исполнение и распределение заказов

2.1 Автоматизированное распределение:
- Платформа должна автоматически назначать заказы курьерам на основе заданных параметров, включая местоположение и загруженность курьеров.

2.2 Ручное назначение:
- Операторы должны иметь возможность вручную распределять заказы между курьерами в случае необходимости.

2.3 Уведомления о распределении:
- После назначения заказа курьеру пользователи и курьеры должны получать уведомления по электронной почте или SMS.

2.4 Отслеживание статусов:
- Пользователи должны иметь возможность отслеживать процесс выполнения своего заказа в режиме реального времени (например, "Создан", "В процессе", "Доставляется", "Доставлено").

2.5 Обновление статусов:
- Курьеры должны иметь возможность изменять статус заказа в зависимости от его текущей стадии (например, "Прибыл", "Вручен").

2.6 Уведомления об изменениях:
- Клиенты должны получать уведомления о каждом изменении статуса их заказа (например, "Доставлено", "Возвращено на доработку").

2.7 Процесс возврата на доработку:
- Пользователи могут вернуть заказ на доработку, если доставка не соответствует их ожиданиям.

### 3. Уведомления и оповещения

3.1 Подтверждение заказа:
- После оформления заказа пользователи и операторы должны получать подтверждающие уведомления по электронной почте с информацией о заказе и уникальным номером.

3.2 Информирование о статусе:
- При каждом изменении статуса заказа пользователи и ответственные курьеры должны автоматически уведомляться.

3.3 Настройка уведомлений:
- У пользователей должна быть возможность настраивать параметры уведомлений, включая частоту и способ получения информации (например, электронная почта или SMS).

### 4. Отчетная система

4.1 Исторический архив заказов:
- Пользователи и администраторы должны обладать возможностью просматривать архив всех заказов, включая как завершенные, так и активные, с опцией фильтрации по различным параметрам, таким как дата и статус.

4.2 Создание отчетов по заказам:
- Платформа должна позволять пользователям формировать отчеты по заказам за заданные временные промежутки, охватывающие данные о статусах, времени выполнения доставок и выполнении условий SLA.

4.3 Мониторинг эффективности:
- Отчеты должны содержать информацию о временных показателях исполнения заказов, соблюдении условий SLA, количестве возвратов и других метриках для оценки качества обслуживания и выявления проблем.

4.4 Экспорт данных:
- Платформа должна поддерживать функцию экспорта отчетов в популярные форматы, такие как PDF и Excel, для удобного хранения и анализа.

### 5. Проверка качества сервиса

5.1 Оценка обслуживания:
- После завершения доставки, пользователям должна быть предоставлена возможность оценить работу курьеров и качество полученных товаров, оставив свои отзывы.

5.2 Сбор обратной связи:
- Система должна собирать и представлять информацию по отзывам и оценкам для оценки уровня удовлетворенности клиентов и выявления направлений для улучшения сервиса.

5.3 Напоминания о предоставлении отзывов:
- В случае, если клиент не оставил отзыв в течение установленного времени (например, 3 дня), система должна автоматически отправить ему напоминание с просьбой оценить доставку.

### 6. Настройки уровней обслуживания (SLA)

6.1 Управление уровнями обслуживания:
- Администраторы должны иметь инструменты для настройки параметров SLA, включая допустимое время отклика и доставки для различных категорий товаров по запросу клиентов.

6.2 Мониторинг выполнения условий SLA:
- Система должна отслеживать время выполнения заказов и оповещать ответственных лиц в случае превышения установленных временных рамок в SLA для обеспечения оперативного реагирования.

### 7. Безопасность и доступ к данным

7.1 Аутентификация пользователей:
- Платформа должна предоставлять безопасную аутентификацию, используя логин и пароль, а также опцию двухфакторной аутентификации для повышения уровня безопасности.

7.2 Управление правами доступа:
- Система должна обеспечивать возможность назначения ролей (например, клиент, администратор, курьер) и определять соответствующие права доступа для обеспечения защиты данных и функциональных возможностей.

7.3 Мониторинг действий пользователей:
- Все действия пользователей, включая создание, редактирование и удаление заказов, должны фиксироваться для поддержки системы безопасности и возможности последующего аудита.

7.4 Защита данных с помощью шифрования:
- Вся информация, передаваемая и хранящаяся в системе, должна быть защищена с использованием современных методов шифрования для обеспечения безопасности клиентских данных.

### 8. Интеграция с другими системами

8.1 Совместимость с почтовыми службами:
- Система должна интегрироваться с почтовыми сервисами для автоматической отправки уведомлений и отчетов клиентам о статусах их заказов.

8.2 Интеграция с ERP-системами:
- Платформа должна предоставлять возможности интеграции с ERP-системами (например, SAP, 1С) для автоматического формирования заказов на основе данных из этих систем, обеспечивая бесперебойный обмен информацией.

### 9. Аналитические инструменты

9.1 Динамические панели управления:
- Система должна представлять пользователям интерактивные панели управления с ключевыми показателями (KPI), такими как среднее время выполнения заказов, количество активных и завершенных заказов, уровень выполнения SLA и клиентская удовлетворенность.

9.2 Визуализация данных:
- Все отчеты и панели управления должны поддерживать графическую визуализацию данных (например, диаграммы и графики) для удобного анализа и оптимизации управленческих решений.
